## New features

- Room categories

  - Admins with the `manage_room_settings` permission can manage room categories in **Admin settings → Rooms → Room Categories**.
  - Categories can be enabled or disabled via a toggle. A second toggle makes selecting a category mandatory when creating a room.
  - Categories are created, renamed, and deleted inline within the settings section. Deletion uses the standard confirmation dialog.
  - When categories are enabled, a **Category** dropdown appears in the dashboard create-room form below the room name. The dropdown shows an "— unspecified —" option by default; if the category is required the field is labelled accordingly.
  - The assigned category is displayed on each room card in the rooms list (right column, between the status pill and the issues-completed count).
  - The rooms list header gains two filter dropdowns — **Status** (replacing the previous row of filter chips) and **Category** — that open and close independently and are mutually exclusive.
  - The create/update/delete operations are logged in the audit log (`room_category.create`, `room_category.update`, `room_category.delete`).

- Rename room

  - Users with the new `rename_room` permission can rename an existing room directly from the room view.
  - A round pencil icon (✎) appears in the room name tile, vertically centred for the full tile height. Clicking it opens a compact centred dialog where the new name can be entered and confirmed with **Save** or dismissed with **Cancel**.
  - The rename action is logged in the audit log (`room.rename`).
  - The new permission `room.rename` is listed under the **Room Management** category in role settings.

- Closed poker room is now locked

  - Once a room reaches `closed` status, no new issues can be added — the add-issue form and the Import from Jira button are hidden, and the backend rejects any attempt to add or start an issue.
  - The table felt replaces the player cards with a centered **Poker closed** label so the state is visually unambiguous.
  - The Close poker button is hidden when the room is already closed.

- Jira assignee picker prioritizes room participants

  - When the **Send to Jira** assignee dropdown opens, Jira users that match a current room participant appear at the top under a **Room participants** label, followed by remaining assignable Jira users below a divider.
  - Matching uses email as the primary signal: the participant's Sprinto email is compared against the Jira user's email address. When Jira does not expose an email for a user, the comparison falls back to display name.
  - Search and filtering work across both sections together.

- Jira import filter builder

  - The two boolean toggles ("Story Points is empty", "Original Estimate is empty") are replaced by a composable condition builder similar to Jira JQL.
  - Each condition row has a **field** dropdown (Story Points / Original Estimate) and an **operator** dropdown (IS EMPTY, IS NOT EMPTY, =, ≠). Selecting = or ≠ reveals a compact numeric value input.
  - Multiple conditions can be chained. An **AND / OR** connector appears between each pair of consecutive rows. Rows beyond the first gain a × button to remove them; **+ Add condition** appends a new row below.
  - The **Import order** field is moved into the Board / Sprint header row so it sits alongside those controls rather than inside the filter section.
  - Jira issues whose original estimate is set to `0m` are now correctly matched by an `originalEstimate = 0` condition. Previously the Jira API omitted `originalEstimateSeconds` for zero-second estimates, causing them to be treated as empty; the parser now falls back to the `originalEstimate` string (e.g. `"0m"`, `"1h 30m"`) and converts it to seconds.

## Bug fixes

- Permission `view_votes_of_others` is now split into two independent role permissions

  - **View Votes of Others** (`view_votes_of_others`) — controls whether a participant can see other players' card values after reveal. Without this permission, other players' cards remain face-down after the round is revealed; the participant's own card still flips normally.
  - **View History** (`view_history`) — controls access to the history panel and the list of previously voted issues. Users with this permission see a **History** button in the controls panel regardless of whether they also hold round-management permissions.
  - Existing roles that held `view_votes_of_others` automatically receive `view_history` on first startup after the update.

- History panel is now accessible for users who only hold the `view_history` permission

  - The controls panel is shown for such users even when they have no other room-management permissions.
  - The queue panel appears alongside it and opens the history overlay when History is clicked.
  - The queue form and queue list remain hidden for users without round-management permissions.

- Controls panel no longer stretches to the full height of the queue panel

  - The panel now sizes to fit its content, so the Delete room button and other controls are always fully visible without overflow.

- Dragging the history timeline scrubber no longer causes text selection

  - Pointer capture is now set on the timeline track element at drag start, which prevents the browser from triggering text-selection behaviour when the pointer moves outside the timeline area.
